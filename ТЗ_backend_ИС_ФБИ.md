# Техническое задание для backend-разработки

## 1. Общие сведения

**Наименование проекта:**  
Backend подсистема информационной системы факультета бизнес-информатики (ИС ФБИ), модуль дашборда декана и связанных ролей.

**Основание для разработки:**  
- ЛР по составлению ТЗ (`ЛР_ТЗ.docx`), требования к структуре документа и контролю приемки;  
- Методические рекомендации (`Техническое задание.pptx`) по ГОСТ 34.602-89, ГОСТ 19.201-78 и SRS;  
- Предметная модель факультета (`Модель факультета.docx`, `Рольи ФБИ.docx`, `декан.docx`);  
- Текущий UI-макет (`dean_dashboard(HTML).html`, `students.html`, `dean_dashboard.js`).

**Цель backend-разработки:**  
Обеспечить серверную часть для хранения, обработки и предоставления данных по учебному процессу, контингенту, ролям и KPI факультета с поддержкой drill-down анализа проблем.

## 2. Назначение системы

Backend предназначен для:
- агрегации данных из учебных сущностей (ОП, УП, РПД, студенты, преподаватели, группы, дисциплины, расписания);
- расчета и публикации KPI декана и других ролей;
- управления ролевым доступом и зонами ответственности;
- предоставления API для дашборда декана и списка студентов;
- ведения календаря событий и заметок;
- фиксации цифрового следа действий пользователей.

## 3. Объект автоматизации и границы

Автоматизируется управление учебной аналитикой и документами факультета в рамках:
- циклов обучения (семестр/год/программа/группа/студент);
- ролевой иерархии (декан, зам. декана, зав. кафедрой, преподаватель, ассистент, студент, администратор);
- ключевых факультетских показателей (успеваемость, отчисления, ВКР, контингент, риски).

В первую очередь покрывается деканский контур, отраженный в текущем интерфейсе.

## 4. Пользователи и роли

### 4.1 Основные роли
- Администратор
- Декан
- Заместитель декана
- Заведующий кафедрой
- Ведущий преподаватель
- Ассистент
- Студент

### 4.2 Принципы ролевой модели
- Одна персона может иметь несколько ролей.
- Одна роль может быть назначена нескольким персонам.
- Поддерживается иерархия подчиненности ролей.
- Роль содержит: права, обязанности, зону ответственности, условия назначения.

## 5. Функциональные требования к backend

## 5.1 Управление данными факультета

Backend должен обеспечивать CRUD и чтение/фильтрацию для сущностей:
- `Person`, `Role`, `RoleAssignment`;
- `Student`, `Teacher`, `Group`, `Direction` (БИСТ, БПИ и др.);
- `EducationProgram` (ОП), `Curriculum` (УП), `Discipline`, `DisciplineProgram` (РПД);
- `AcademicYear`, `Semester`, `ExamSession`;
- `EnrollmentMovement` (зачисление/отчисление/академ. отпуск);
- `ScheduleClass`, `ScheduleExam`;
- `FacultyEvent`, `CalendarNote`.

## 5.2 KPI и аналитика декана

Backend должен рассчитывать и отдавать:
- общее число студентов;
- средний балл по факультету;
- долю студентов без задолженностей;
- уровень отчислений;
- KPI-цели декана (факт/цель/статус);
- сравнение направлений подготовки;
- топ проблемных групп;
- сигналы рисков и предупреждения;
- контингент и движение.

Расчеты выполняются по выбранным фильтрам:
- учебный год;
- семестр;
- направление;
- уровень детализации (факультет/направление/группа/студент).

## 5.3 Drill-down и сигналы

Backend должен поддерживать сценарий drill-down:
- от KPI факультета к направлению;
- от направления к группе;
- от группы к студентам/дисциплинам с вкладом в проблему.

Должны формироваться сигналы (risk alerts) при настраиваемых правилах, например:
- превышен порог отчислений;
- высокая доля задолженностей по дисциплине/группе;
- отрицательная динамика по направлению.

## 5.4 Личный кабинет и рабочие пространства

Backend должен поддерживать:
- персональный кабинет с дашбордом роли;
- список доступных подпространств и документов;
- операции чтения/редактирования документов по зоне ответственности;
- сообщения/уведомления по событиям и заданиям.

## 5.5 Календарь и заметки

По UI-макету должны быть реализованы:
- события факультета по датам;
- заметки к дате с операциями создать/изменить/удалить;
- хранение заметок на сервере (вместо in-memory на фронте);
- аудит изменений заметок.

## 5.6 Данные студентов

Для страницы списка студентов backend предоставляет:
- пагинированный список студентов;
- серверную сортировку по полям (ID, ФИО, группа, курс, год набора, средний балл, задолженности, риск);
- фильтрацию по году набора и направлению;
- расчет/выдачу статуса риска.

## 5.7 Аудит и цифровой след

Каждое значимое действие фиксируется:
- кто выполнил;
- когда;
- над какой сущностью;
- старое и новое значение;
- источник действия (UI/API/integration).

## 6. Нефункциональные требования

## 6.1 Производительность
- Ответы агрегатных API дашборда: не более 1.5 сек при типовой нагрузке.
- Ответы справочных API: не более 500 мс.
- Поддержка не менее 200 одновременных активных пользователей.

## 6.2 Надежность и доступность
- Доступность backend: не ниже 99.5% в учебный период.
- Транзакционность операций записи критичных данных.
- Резервное копирование БД: ежедневно, хранение минимум 30 дней.

## 6.3 Безопасность
- Аутентификация (JWT или сессионная, по согласованию).
- RBAC на уровне API-методов и данных.
- Защита персональных данных студентов и преподавателей.
- Логирование попыток несанкционированного доступа.
- Шифрование трафика (HTTPS/TLS).

## 6.4 Сопровождаемость
- Чистая модульная архитектура (слои: API, application, domain, infrastructure).
- Принципы SOLID в бизнес-логике.
- Расширяемость для новых ролей/показателей/KPI без изменения ядра.

## 7. Внешние интерфейсы

REST API (базово, версия `v1`) с группировкой:
- `/auth/*`
- `/reference/*` (справочники)
- `/students/*`
- `/dashboard/dean/*`
- `/analytics/*`
- `/calendar/*`
- `/roles/*`
- `/documents/*`
- `/audit/*`

Формат обмена: JSON UTF-8.  
Документирование API: OpenAPI 3.0.

## 8. Требования к данным и БД

Рекомендуемая БД: PostgreSQL.

Минимальные таблицы:
- users, persons, roles, role_assignments;
- students, teachers, groups, directions;
- education_programs, curriculums, disciplines, discipline_programs;
- academic_periods, enrollments, expulsions, academic_leaves;
- grades, debts, kpi_targets, kpi_snapshots;
- events, calendar_notes;
- documents, document_access;
- audit_log, alerts.

Обязательные требования:
- внешние ключи и ограничения целостности;
- индексация фильтров (год, семестр, направление, группа, роль);
- soft delete для журналируемых сущностей.

## 9. Интеграции

На первом этапе допускается автономный контур.  
На втором этапе предусмотреть интеграции (через адаптеры):
- импорт контингента и приказов;
- импорт расписаний;
- импорт учебных планов и РПД;
- экспорт отчетов для руководства.

## 10. Этапы реализации

1. Проектирование доменной модели и БД.  
2. Реализация auth + RBAC.  
3. Реализация справочников и ядра учебных сущностей.  
4. Реализация API дашборда декана и аналитики KPI.  
5. Реализация API списка студентов и фильтрации/сортировки.  
6. Реализация календаря и заметок с аудитом.  
7. Тестирование, оптимизация, документация OpenAPI.  

## 11. Требования к тестированию и приемке

Приемка backend выполняется по критериям:
- все endpoint'ы покрыты интеграционными тестами;
- критичная бизнес-логика KPI покрыта unit-тестами;
- проверена ролевая изоляция доступа;
- подтверждена корректность drill-down цепочек;
- API соответствует согласованному контракту OpenAPI.

Минимальное покрытие тестами: 70% для business/domain слоев.

## 12. Артефакты результата

Backend-разработчик должен предоставить:
- исходный код сервиса;
- миграции БД;
- OpenAPI спецификацию;
- коллекцию запросов для smoke-тестов;
- инструкцию запуска;
- описание принятых архитектурных решений;
- краткий регламент расширения ролей и KPI.

## 13. Риски и ограничения

- Требования к ролям в исходных материалах частично неполные; уточняются на этапе проектирования.
- Данные в текущем frontend являются демонстрационными и требуют замены на реальные источники.
- Нормативные требования ГОСТ/ISO трактуются как рамка структуры ТЗ и критериев качества.

## 14. Перспективы развития

- Подключение ИИ-ассистента роли для рекомендаций и анализа причин отклонений.
- Расширение событийной модели для управления заданиями и дедлайнами.
- Поддержка прогнозирования рисков отчисления и качества обучения.
